# SDLC-Dev: 系统研发阶段标准化技能体系

本技能模块旨在为软件开发生命周期的【研发阶段】提供全方位的标准化规范和最佳实践指南。涵盖需求分析、架构设计、技术选型、编码规范、代码审查、性能优化等核心环节。

## 1. 多维度适配指南 (组合调用矩阵)

根据您的项目特性，应用以下推荐配置：

### 按项目类型适配
- **Web应用**: 重点关注前后端分离架构、XSS/CSRF防护、首屏加载优化（SSR/SSG）。
- **移动应用**: 重点关注离线缓存、API版本控制、弱网优化、多端适配（React Native/Flutter）。
- **大数据系统**: 重点关注流批一体架构、数据血缘、列式存储选型、计算资源隔离。
- **AI平台**: 重点关注模型版本管理、GPU资源调度、推理延迟优化、数据隐私计算。

### 按技术栈适配
- **Java/Spring**: 强制使用CheckStyle、Alibaba Java Coding Guidelines；采用Maven/Gradle依赖管理。
- **Python/Django**: 强制使用PEP8、Black格式化；采用Poetry/Pipenv环境隔离；关注GIL对并发的影响。
- **Node.js**: 强制使用ESLint、Prettier；采用TypeScript增强类型安全；关注事件循环阻塞问题。
- **Go**: 遵循Effective Go；使用gofmt、golangci-lint；重点关注Goroutine泄露和并发安全。
- **.NET**: 遵循C# Coding Conventions；使用Roslyn Analyzers；依赖NuGet。

### 按团队规模适配
- **初创团队 (1-10人)**: 采用单体或模块化单体架构，轻量级敏捷开发，主干开发模型（Trunk-based Development）。
- **中型团队 (11-50人)**: 引入微服务演进，GitFlow或GitHub Flow，严格的CR（Code Review）制度，建立基础组件库。
- **大型组织 (50人+)**: 领域驱动设计（DDD），多体仓（Polyrepo）或巨型仓（Monorepo）管理，跨团队API契约（Swagger/Protobuf）。

### 按合规要求适配
- **等保 (2.0)**: 强制审计日志记录，密码复杂度与加密存储（国密算法SM2/SM3/SM4优先），强制权限最小化设计。
- **GDPR**: 实施数据脱敏（PII屏蔽），提供数据被遗忘权（物理删除/不可逆伪匿名化）的API设计，跨境数据合规检查。
- **SOX**: 职责分离（开发与运维分离），强制的代码变更审批流，所有的架构变更需留存追溯文档。

## 2. 核心技能模块定义

### 2.1 需求分析与架构设计 (Architecture & Design)
*   **实施步骤**:
    1. 收集业务需求，识别功能与非功能需求（性能、可用性、扩展性）。
    2. 领域建模（DDD），划分限界上下文，定义聚合根。
    3. 绘制C4模型（系统上下文、容器、组件、代码级别）。
    4. 架构评审会议（ADR记录决策）。
*   **工具推荐**: Draw.io, PlantUML, Structurizr, Xmind.
*   **质量门禁指标**: 架构评审通过率100%，非功能需求（如QPS/SLA）明确量化，ADR（架构决策记录）归档。
*   **风险控制措施**: 引入架构原型（PoC）验证关键技术可行性，避免过度设计。
*   **交付物模板**: 《系统架构设计说明书》(SAD)，架构决策记录（ADR）。
*   **验收标准**: 架构图清晰无歧义，关键接口契约已定义，上下游依赖方已确认。

### 2.2 技术选型与框架搭建 (Technology Selection)
*   **实施步骤**:
    1. 明确选型需求（生态、性能、社区活跃度、团队熟悉度）。
    2. 竞品对比矩阵分析。
    3. 编写脚手架/模板项目，集成基础中间件（日志、配置、监控）。
*   **工具推荐**: 技术雷达（ThoughtWorks），Slant.co对比，Yeoman/Plop脚手架生成器。
*   **风险控制措施**: 避免引入不再维护的“僵尸”开源项目，商业版需评估License合规性（GPL传染性风险）。
*   **交付物模板**: 《技术选型对比分析报告》。

### 2.3 编码规范与代码审查 (Coding Standards & Code Review)
*   **实施步骤**:
    1. 统一IDE配置（EditorConfig）。
    2. 提交前本地Hook检查（Pre-commit/Husky）。
    3. 提交PR/MR，至少1-2名同级别或高级别工程师进行Review。
    4. 关注：逻辑正确性、安全漏洞、可读性、可测试性。
*   **工具推荐**: SonarQube, GitHub Actions (ReviewBot), Gerrit, Husky, CheckStyle/ESLint/GolangCI-Lint。
*   **质量门禁指标**: 严重（Blocker/Critical）代码异味=0，技术债务率<5%，代码重复率<5%。
*   **交付物模板**: PR/MR 描述模板，代码审查检查单（Checklist）。
*   **验收标准**: CI流水线静态扫描通过，所有Review意见已解决或转化为技术债务卡片。

### 2.4 性能优化基础 (Performance Optimization)
*   **实施步骤**:
    1. 建立性能基准（Baseline）。
    2. 遵循“二八原则”，定位瓶颈（CPU、内存、I/O、网络）。
    3. 实施优化：数据库索引、多级缓存（本地+分布式）、异步处理（MQ）、批量处理。
*   **工具推荐**: JProfiler, Arthas, pprof, Chrome DevTools, JMH。
*   **质量门禁指标**: 接口P99响应时间达标，复杂查询命中索引率>95%。
